# REQUIREMENTS.md — レトロゲーム実況の相棒AI（仮称：あいきょう / AI Kyou）

## 0. このドキュメントの読み方（Claude Code向け）

- これは **Phase 1（MVP）の実装指示書**です。
- 「やらないこと」リストは**強い制約**として扱ってください。書いていない機能は作らないでください。
- Phase 2 以降は方向性のメモであり、**先読みして実装しないでください**。MVPが動いてから別途要件を切ります。

---

## 1. 目的・背景

- 開発者が一番のヘビーユーザー（**自分専用ツール**）。他人に使わせる前提なし。
- 最終的な成果物は Web サービスそのものではなく、**「AIと会話しながら昔のゲームをする」YouTube動画**。
- このアプリは「動画を撮るための相棒」。アプリの体験設計より、**会話が成立して続きが繋がること**を最優先する。

### ゴール（MVPの完成条件）
1. ファミコン版ドラゴンクエスト3をプレイしながら、AIと自然に会話できる。
2. プレイを中断・再開しても、AIが**前回の続き（パーティ・現在地・次の目標）を覚えている**。
3. これらの操作が、開発者ひとりが PC で完結できる。

---

## 2. コンセプト

- 攻略サイトを見れば答えは分かる。**あえてAIと会話しながら進める**のが面白さ。
- AIは「答え自動販売機」ではなく「**少し詳しい先輩ゲーマー（相棒）**」として振る舞う。
- RPGは長時間プレイ＝複数セッションになるため、**会話の外に「プレイスルーの状態（state）」を持たせる**ことで継続性を担保する。これがこのアプリの核。

---

## 3. 技術スタック

| 領域 | 採用 |
|------|------|
| フレームワーク | Next.js (App Router) + TypeScript |
| スタイル | Tailwind CSS |
| DB | Supabase（テーブル1つ、JSONB中心） |
| AI | Anthropic API（Claude）を **Next.js の API Route 経由**で呼ぶ |
| モデル | **Claude Haiku 4.5**（モデルID: `claude-haiku-4-5-20251001`）。MVPは全エンドポイントこれで統一 |
| APIキー | サーバ側の環境変数 `ANTHROPIC_API_KEY` のみ。クライアントに出さない |
| SDK | `@anthropic-ai/sdk` を使用 |

> 自分専用なので **Supabase の RLS・認証は設定しない**（ローカル/限定環境で動かす前提）。

---

## 4. スコープ

### 4.1 MVPでやること（Phase 1）

1. **プレイスルー一覧 / 新規作成**：ゲームタイトル＋初期stateで1件作る。一覧から選んで開く。
2. **セッション画面**
   - 画面上部に「**前回までのあらすじ**」（現在のstateを人が読める形で表示）。
   - 中央にチャットUI（テキスト入力・送信、AI応答の表示）。
   - 「**セッション終了**」ボタン → 後述のstate更新を実行して保存。
3. **AIとの会話**：送信時に「現在のstate＋攻略ナレッジ」をシステムプロンプトに載せて応答を返す。
4. **state更新（継続性の心臓部）**：セッション終了時に、会話内容を踏まえた**新しいstate（JSON）をAI自身に生成させて保存**する。

### 4.2 MVPでやらないこと（重要・先読み実装禁止）

- ❌ ユーザー認証・ログイン・マルチユーザー
- ❌ 音声入力（STT）・音声合成（TTS）
- ❌ AIのキャラクター/ペルソナ設定UI
- ❌ YouTube用の「冒険ログ」整形・書き出し機能
- ❌ 複数ゲームへの汎用化・攻略ナレッジの管理画面
- ❌ RAG・ベクトルDB・埋め込み検索（攻略はテキスト直渡しで十分）
- ❌ 凝ったデザイン・アニメーション・レスポンシブ最適化（PCで動けばよい）
- ❌ 会話履歴の全文DB保存（state さえ残ればよい。下記参照）

## 5. データモデル

Supabase テーブルは **`playthroughs` 1つだけ**。

```
playthroughs
- id            uuid (pk, default gen_random_uuid())
- title         text          // 例: "ドラゴンクエスト3"
- game_version  text          // 例: "ファミコン版(FC)" ← 必須・重要
- state         jsonb         // 下記の構造
- created_at    timestamptz default now()
- updated_at    timestamptz default now()
```

### `state` の構造（JSONB）

```jsonc
{
  "party": [
    { "name": "ゆうしゃ", "job": "勇者",   "level": 5 },
    { "name": "ガンダ",   "job": "戦士",   "level": 5 },
    { "name": "ぶとう",   "job": "武闘家", "level": 4 },
    { "name": "ホイミン", "job": "僧侶",   "level": 4 }
  ],
  "location": "アリアハン周辺",
  "progress": "ルイーダの酒場でパーティ編成。レベル上げ中。",
  "next_goals": ["レベル5前後まで上げる", "ロマリアへ向かう"],
  "notes": "MP管理がきつい。僧侶のMP温存中。"
}
```

- フィールドは固定スキーマにせず **緩く扱う**（AIが追記しても壊れないように）。
- 必須として扱うのは `party / location / next_goals` の3つ。

## 6. 画面・機能仕様（MVP）

### 6.1 プレイスルー一覧（`/`）
- `playthroughs` を一覧表示（title・location・updated_at）。
- 「新規作成」：title と game_version を入力し、空に近い初期stateで作成 → セッション画面へ。

### 6.2 セッション画面（`/play/[id]`）
- **あらすじ表示**：現在の `state` を読みやすく整形して表示（パーティ／現在地／次の目標／メモ）。
- **チャット**：
  - 入力欄＋送信。送信で `/api/chat` を叩く。
  - 会話履歴は**そのセッション中だけクライアントのメモリに保持**（リロードで消えてよい。継続性はstateが担う）。
  - HTMLの `<form>` ではなく onClick / onChange で処理する。
- **セッション終了**ボタン：`/api/end-session` を叩き、返ってきた新stateを保存 → あらすじを更新。

## 7. AI連携仕様

### 7.1 会話：`POST /api/chat`
- 入力：`{ playthroughId, messages }`（messagesはそのセッションの会話履歴）。
- サーバ側でDBから `state` を取得し、**システムプロンプト**を組み立てて Claude に投げる。
- 出力：AIのテキスト応答をそのまま返す。

**システムプロンプトの構成（最重要）**
```
あなたは「{title}（{game_version}）」を一緒に遊ぶ、少し詳しい先輩ゲーマー（相棒）です。

【絶対のルール】
- 必ず「{game_version}」の仕様で答えること。他の版（SFC/GBC/スマホ/HD-2Dリメイク等）の
  情報を混ぜないこと。職業・呪文・ダンジョン構成は版で異なるため、ここを間違えない。
- 答えを一方的に全部言うのではなく、会話相手として、聞かれたことに的確に答える。

【現在の冒険の状況】
{state を読みやすく整形したもの}

【攻略の参考知識（この版の情報）】
{攻略ナレッジ markdown を丸ごと貼り付け}
```

> モデルは `claude-haiku-4-5-20251001`（Haiku 4.5）。速くて安く、会話のテンポ重視のこの用途に最適。

### 7.2 state更新：`POST /api/end-session`
- 入力：`{ playthroughId, messages }`。
- サーバ側で「現在のstate」＋「今回の会話全文」を渡し、**新しいstateだけをJSONで返す**ようClaudeに依頼。
- プロンプト指示例：
  ```
  以下は前回までの状態と、今回のプレイ会話です。
  今回の進展を反映した「新しい状態」を、前回と同じJSON構造で出力してください。
  出力はJSONのみ。前置き・説明・コードフェンスは一切付けないこと。
  ```
- **安全にパース**する（```json フェンスが付いてきたら除去 → JSON.parse → 失敗時は旧stateを維持しエラー表示）。
- パース成功したら `state` と `updated_at` をDB更新。

> **モデルについて**：MVPはここも Haiku 4.5。もし state のJSON出力が崩れる頻度が高ければ、**この `/api/end-session` だけ** Sonnet 4.6（`claude-sonnet-4-6`）に上げてよい（1セッションに1回しか呼ばないのでコスト影響は誤差）。chat 側は Haiku のままでよい。

## 8. 攻略ナレッジの渡し方（MVP）

### 大前提（重要）
- **Claude API は渡されたURLを開かない**。URLのリストを渡してもAIは中身を読めない。
- AIに「認識させる」とは、**該当テキストをシステムプロンプトに含める**こと。これ以外の方法はMVPでは取らない。
- リトリーバル（現在地等に応じて必要なファイルだけ選ぶ仕組み）は**作らない**。MVPは**全ファイルを連結して丸ごと渡す**。

### 知識ファイルの構成
- 攻略ナレッジは **`/knowledge/dq3-fc/` フォルダ内の複数の markdown ファイル**（職業・町・ダンジョン・基本情報などに分割。合計48ファイル程度）。
- 加えて **`_ai-notes.md`**：FC版の注意（性格システム無し等）／AIの振る舞いメモ／裏技をまとめたメタ情報ファイル。
- 分割はあくまで**人間の管理のため**。アプリ側の扱いは下記の通り「全部読んで連結する」だけで、1枚のときと実質同じ。

### ファイル配置・命名（実装の前提）
- 置き場所は**プロジェクト直下の `knowledge/dq3-fc/`**。`public/` には**置かない**（公開URLで取得可能になるため）。`src/` 配下でもなくルート直下でよい。
- サーバ側（API Route）から `path.join(process.cwd(), 'knowledge', 'dq3-fc')` で読む。クライアントには含めない。
- **ファイル名はASCII（英数字・ハイフン）**にする（例: `job-senshi.md`, `town-romaly.md`, `dn-izanai.md`）。日本語ファイル名はWSL2/Windows/Git間の文字コード正規化の食い違いで事故るため使わない。**日本語のタイトルはファイル内の見出し（1行目の `# …`）として保持**する（AIは見出しで内容を識別する）。

### アプリでの扱い
- 起動時（またはセッション開始時）に **`knowledge/dq3-fc/` 内の .md を全て読み込んで1つの文字列に連結**し、システムプロンプトに含める。
- ファイル選別は行わない（全件連結）。
- **読み込み順はファイル名で決定的にソートする**（`readdir` の返却順はOS依存なので必ずソートする）。順序が毎回同じでないとプロンプトキャッシュが効かない。`_ai-notes.md` を先頭に固定する。
- 連結結果はメモリにキャッシュし、毎リクエストでファイルを読み直さない。

### プロンプトキャッシュ（重要・必須に近い）
- 連結後の知識は約4万トークン規模になる見込み。毎ターンそのまま再送するとコストがかさむため、**プロンプトキャッシュを使う**。
- プロンプトの構造を以下の順に固定する：
  1. **システム指示 ＋ 連結した攻略ナレッジ**（毎回不変）← **ここをキャッシュ対象**にする
  2. 現在の `state`（毎セッション更新）
  3. そのセッションの会話履歴（毎ターン増える）
- 上記1を**プロンプトの先頭に固定**することでキャッシュヒットさせ、2回目以降の入力コストを大幅に下げる。

> 補足：知識ファイルは外部攻略サイト由来の本文を含むため、**サーバ側の入力データとしてのみ利用**し、ファイル自体を公開・再配布しない。表示・配信されるのはAIの応答であって生データではない。
> 現在地に応じて関連ファイルだけを引く最適化（リトリーバル）が必要になったら Phase 2 で別途要件を切る。

## 9. Phase 2以降（方向性メモ・実装しない）

- 攻略ナレッジのリトリーバル：現在地に応じて関連情報を引く検索の仕組み（サイト全体規模の情報を扱いたくなった場合）
- 音声：自分の声を文字起こし → AI応答 → 読み上げ（会話のテンポ＝動画の質）
- AIのペルソナ命名・口調設定
- 「今日の冒険ログ」をYouTube用に整形・コピーできる出力
- 複数ゲーム対応／攻略ナレッジの差し替えUI
- 会話履歴の保存・ふりかえり

## 10. 補足・非機能

- 動作環境：開発者のPC（ローカル）で動けばよい。デプロイ・公開は不要。
- パフォーマンス／同時接続の考慮不要（1人）。
- エラー時は握りつぶさず画面に出す（自分でデバッグするため）。
- 長時間プレイ対策：会話履歴はセッション単位で区切る。**セッションをまたぐ継続性はstateのみが担う**（履歴を無限に積まない）。
